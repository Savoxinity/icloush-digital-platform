# iCloush 智慧工厂——Phase031 零售堡垒与私域转化矩阵 V3.1.0

## 版本概述

本次 Phase031 明确终止 B2B 企业入驻与 Stripe 方向的继续开发，全面转向 **iCloush LAB. 零售堡垒与私域转化矩阵**。本轮工作的核心不是继续堆叠管理后台复杂度，而是把现有数字基建重新收束成一个更有掠夺力的零售前台：用 Monolithic Hero 首屏建立品牌视觉锤，用官网购物袋与零售下单 JSON API 承接交易意图，再用外部入口桥接和二维码矩阵把淘宝、天猫与小程序渠道重新组织成可转化的私域/公域联动网络。

## 本次更新内容

| 模块 | 更新内容 | 结果 |
| --- | --- | --- |
| 首页 Hero | 首页重构为 **iCloush LAB.** 超大字重主视觉，加入 `// 3026 ORBITAL JEWELER //` 微缩副标题、巨物悬浮背景层与唯一 `/showroom` 锋利箭头入口 | 首屏不再承担说明书角色，而是承担“视觉锤”与心理掠夺入口 |
| 零售 Showroom / PDP | 在既有中文化 Showroom 与 PDP 基础上保留深空方尖碑语言，并把购物袋、SKU 选择与 `EXTERNAL ACCESS / 外部入口` 悬浮层纳入主链路 | 官网前台从展示页升级为可承接购买动作的零售矩阵 |
| Cart 与下单链路 | 前台购物袋透传真实 `productId / skuId`，接入零售下单 mutation，并在界面中轮询订单状态，显示 `TRANSACTION SUCCESSFUL` 等反馈 | 官网已具备真实订单草稿生成与支付状态反馈骨架 |
| API Gateway | 在网关侧新增真正的 `POST /api/orders/retail` 与订单状态查询 REST 接口，对外返回标准化支付参数 envelope | 为后续网页、H5 或外部页面调用零售支付 JSON API 提供统一壳层 |
| Admin 商品工作台 | 商品表单新增淘宝短链、小程序路径、微信/支付宝二维码 URL 等维护字段，并序列化进 specs 结构 | 无需额外迁移数据库即可先完成管理侧配置入口 |
| 前台二维码桥接 | web-b2b 前台新增对零售桥接元数据的解析，优先展示后台录入的淘宝/天猫链接与二维码图像；未配置时则保留 reserved 态占位 | PDP 的外部入口从静态文案升级为可被后台驱动的转化桥接层 |
| 测试 | 补充并运行 web-b2b、admin 与 api-gateway 的 Vitest 覆盖，其中新增了零售 REST 端点的 HTTP 级测试 | 当前零售堡垒主干链路已有可回归的前后端测试基线 |

## 验证结果

本轮完成后，`apps/web-b2b/src/App.test.tsx` 已通过，覆盖 Monolithic Hero、中文 Showroom、PDP 外部入口与后台二维码元数据展示；`apps/admin/src/App.test.tsx` 已通过，覆盖 Admin 商品工作台新增零售桥接字段；`apps/api-gateway/tests/gateway.test.ts` 与新增的 `gateway.rest.test.ts` 已通过，验证了零售 REST 端点与既有网关行为的可用性。由此可确认，当前零售堡垒版本已经形成可审阅、可回归、可继续演进的交付基线。

## 交付备注

需要说明的是，当前支付 JSON API 仍然输出的是 **标准化占位参数结构**，前台支付状态轮询也仍以现有订单状态为准，尚未完成真实微信支付/支付宝 webhook 的正式闭环。因此，本版本的价值在于把“视觉锤 + 购物袋 + 下单壳层 + 外部入口桥接 + Admin 可配置二维码”全部串起来，为下一阶段接入真实支付证书、商户号、公钥和异步回调扫清结构性障碍，而不是宣称真实收款已经完全上线。

# iCloush 统一数字平台 PRD

作者：**Manus AI**  
版本：V0.1 讨论稿  
日期：2026-04-08

## 一、项目定义

本项目旨在建设一个 **统一数字平台**，整合自有商城系统与三个独立官网，分别服务于 B2B 采购转化、品牌信任建立、行业解决方案展示与高端酒店奢护服务咨询。该平台不是四个独立网站的简单集合，而是一套共享账户、商品、订单、内容与 SEO 能力的多站点系统。

根据 B2B 电商建设的通行做法，企业采购平台的关键不只是商品展示，而是是否能够支撑复杂采购流程、账户管理、订单协同、支付与后续复购体验。[1] 同时，Stripe Checkout 可在较低开发复杂度下提供安全、响应式且支持多支付方式的支付体验；而 sitemap 作为结构化抓取文件，应从一开始就纳入多站点 SEO 架构设计。[2] [3]

## 二、项目目标

本项目的目标分为商业目标、用户目标和系统目标三个层面。

| 目标层级 | 目标描述 | 对应结果 |
| --- | --- | --- |
| 商业目标 | 建立统一的线上品牌与交易平台，承接 B2B 客户采购、品牌展示与销售线索获取 | 提升线上获客能力、客户转化能力与品牌矩阵协同能力 |
| 用户目标 | 让企业客户能够快速了解产品、解决方案与服务，并顺利完成采购或咨询 | 降低信息获取成本，缩短决策路径，提高询盘与下单效率 |
| 系统目标 | 建立统一的数据底座和后台体系，减少重复建设与后续维护成本 | 商品、订单、客户、内容、SEO 能力统一管理 |

## 三、项目范围

本期 PRD 覆盖以下范围：统一平台门户、自有商城系统、用户系统、iCloush LAB. 官网、环洗朵科技官网、富朵朵 iCloush Care 官网、统一后台管理系统、多站点路由架构、SEO 基础设施与响应式体验。

本期不默认纳入复杂 ERP 对接、企业审批流、合同管理、账期管理、多语言、多币种、跨境税务、经销商返利和高级 BI 分析等能力。这些内容将在数据模型与系统架构中预留扩展位，但不作为首期必须交付范围。[1] [2]

## 七、功能需求详述

### 7.1 商城系统

商城系统应围绕 B2B 客户采购效率设计，而不是以单纯零售冲动转化为唯一逻辑。[1] 首期功能要求如下。

| 模块 | 功能点 | 说明 | 优先级 |
| --- | --- | --- | --- |
| 商品展示 | 商品列表 | 展示产品图、名称、简介、核心规格、价格或“咨询价格”状态 | P0 |
| 商品展示 | 分类浏览 | 支持按产品分类、应用场景、行业场景进行浏览 | P0 |
| 商品展示 | 搜索与筛选 | 支持关键词搜索与基础筛选 | P1 |
| 商品详情 | 详情页展示 | 包含卖点、参数、规格、适用场景、使用建议、包装信息 | P0 |
| 购物车 | 加入购物车 | 支持数量调整、删除商品、继续购物 | P0 |
| 结算 | 订单确认 | 展示收货信息、商品明细、金额摘要 | P0 |
| 支付 | Stripe 支付 | 调起支付会话并完成支付 | P0 |
| 订单 | 订单查看 | 查看订单状态、金额、商品明细与支付状态 | P0 |
| 订单 | 再次购买 | 从历史订单快速复购 | P1 |
| 订单 | 取消/售后状态展示 | 首期可做基础展示，不承诺完整售后流程 | P2 |

#### 商城核心流程

商城的标准流程应为：用户浏览商品、进入详情页、加入购物车、提交订单、发起支付、支付完成、订单状态更新、用户在个人中心查看历史订单。支付状态应由服务端通过 Stripe webhook 异步确认，而不应只依赖前端支付成功页面返回。[2]

### 7.2 用户系统

用户系统主要服务 B2B 客户注册、登录、企业身份沉淀与订单留存。

| 模块 | 功能点 | 说明 | 优先级 |
| --- | --- | --- | --- |
| 账户体系 | 注册 | 支持企业客户注册，首期可采用基础注册表单 | P0 |
| 账户体系 | 登录/退出 | 支持登录、退出与登录状态维持 | P0 |
| 用户资料 | 个人中心 | 展示姓名、联系方式、企业信息等基础资料 | P0 |
| 订单历史 | 订单列表 | 显示历史订单、支付状态、创建时间 | P0 |
| 订单历史 | 订单详情 | 查看历史订单的商品与金额信息 | P0 |
| 企业信息 | 企业资料维护 | 为后续企业分层、协议价、审批流预留扩展位 | P1 |

iCloush LAB. 站点的核心任务是建立研发能力、技术可信度与品牌认知。其内容应偏品牌高度与技术深度，而不是直接重复商城商品页。

| 页面 | 核心内容 | 主转化 |
| --- | --- | --- |

### 7.5 富朵朵 iCloush Care 官网

富朵朵 iCloush Care 站点应突出酒店奢护洗涤服务的高端感、专业流程感和合作信任感。

| 页面 | 核心内容 | 主转化 |
| --- | --- | --- |

## 八、支付方案

Stripe Checkout 支持托管或嵌入式支付流程，能够减少支付页自研复杂度，且内置移动端适配、PCI 合规与多支付方式支持。[2] 因此，首期建议采用以下支付链路。

| 步骤 | 系统动作 | 责任方 |
| --- | --- | --- |
| 1 | 用户在商城确认订单并点击支付 | 前端 |
| 2 | 服务端创建 Stripe Checkout Session | 后端 |
| 3 | 前端跳转到 Stripe 支付页 | 前端 |
| 4 | 用户完成支付 | Stripe 托管页 |
| 5 | Stripe 通过 webhook 回调支付成功事件 | Stripe -> 后端 |
| 6 | 后端更新订单状态并记录支付结果 | 后端 |
| 7 | 用户返回站内查看支付结果与订单详情 | 前端 |

## 九、SEO 与内容发布需求

Google 将 sitemap 定义为向搜索引擎说明页面及其关系的结构化文件，尤其对结构复杂或较大的站点更有帮助。[3] 因此，本项目应在首期就建立多站点 SEO 基础能力。

| SEO 能力 | 要求 | 优先级 |
| --- | --- | --- |
| 独立 meta | 每个站点、每个页面支持独立 title 与 description | P0 |
| 独立 slug | 页面与内容可设置 SEO 友好链接 | P0 |
| sitemap | 为商城、LAB、环洗朵科技、Care 生成独立 sitemap | P0 |
| sitemap index | 统一索引所有子 sitemap | P1 |
| 站内链接结构 | 页面之间存在清晰的导航与互链逻辑 | P0 |
| Open Graph | 页面支持基础 OG 信息 | P1 |
| robots/收录控制 | 为未发布页面控制可见性与抓取 | P1 |

## 十、响应式设计需求

Stripe Checkout 本身具备响应式移动端设计能力，但商城和三个官网的整体前台体验仍需在平台层完成完整适配。[2] 本项目要求桌面端与移动端均可顺畅浏览、查询、下单和提交咨询。

| 终端 | 设计要求 |
| --- | --- |
| 桌面端 | 强调信息完整度、导航效率和内容层级清晰 |
| 平板端 | 保持主要布局结构，优化留白和卡片栅格 |
| 移动端 | 优先保证导航可用性、表单易填写、支付流程顺畅 |

## 十一、视觉设计方向

由于当前未指定品牌视觉风格，本项目建议采用“统一设计系统 + 站点主题差异化”的策略，以兼顾平台一致性与品牌独立性。

| 站点 | 建议视觉方向 |
| --- | --- |
| 商城系统 | 清晰、理性、专业，强调目录感与采购效率 |
| iCloush LAB. | 科研感、未来感、极简技术气质 |
| 环洗朵科技 | 稳健、专业、工业科技感 |
| 富朵朵 iCloush Care | 高端、温和、洁净、酒店服务审美 |
| 后台管理 | 克制、清晰、高信息密度的 Dashboard 风格 |

| 类型 | 要求 |
| --- | --- |
| 性能 | 首页、分类页、详情页需保证良好的首屏加载体验 |
| 安全 | 支付通过 Stripe 托管流程完成，订单状态以服务端为准 [2] |
| 扩展性 | 允许后续扩展更多品牌站点、更多支付与报价方式 |
| 可维护性 | 统一后台管理，避免多套重复配置 |
| 可运营性 | 页面、案例、方案与联系方式支持后台配置 |

## 十五、MVP 建议边界

为了尽快上线首个可用版本，建议将 MVP 控制在“统一架构成立、核心交易可跑通、三个官网可展示、后台可管理”的范围内。

| 模块 | MVP 包含 | 暂缓项 |
| --- | --- | --- |
| 商城 | 产品展示、分类、购物车、订单、Stripe 支付 | 协议价、审批流、账期结算 |
| 用户系统 | 注册、登录、个人中心、订单历史 | 多层级采购角色 |
| 官网 | 每站 4 至 5 个核心页面 | 新闻中心、复杂内容专题 |
| 后台 | 商品、订单、客户、内容基础管理 | 高级 BI、工作流自动化 |
| SEO | 独立 meta、独立 sitemap、基础站内链接 | 复杂结构化数据增强 |

## 十六、当前待确认事项

进入正式开发前，建议你确认以下关键决策，因为这些会直接影响 IA、数据库模型与视觉稿方向。

| 待确认项 | 选项或说明 |
| --- | --- |
| 平台总入口是否需要单独首页 | 建议需要，用于统一品牌矩阵导流 |
| 正式上线域名策略 | 子域名为主，还是部分品牌独立域名 |
| 商城是否首期显示价格 | 可选“公开价格”或“咨询价格/登录后可见” |
| 支付范围 | 首期是否只做 Stripe 一次性支付 |
| 用户注册字段 | 是否必须采集企业名称、联系人、电话、行业等 |
| 官网咨询方式 | 仅表单，还是加即时沟通入口 |
| 后台内容管理粒度 | 仅文章级，还是页面模块级配置 |

## 十七、验收标准草案

| 模块 | 验收标准 |
| --- | --- |
| 多站点架构 | 商城与三个官网拥有独立入口、独立导航、独立页面结构 |
| 用户系统 | B2B 客户可注册、登录并查看订单历史 |
| 商城 | 可完成商品浏览、购物车、下单与支付闭环 |
| Stripe 支付 | 支付完成后订单状态由服务端正确更新 |
| 官网内容 | 三个官网核心页面均可独立展示与配置 |
| 后台 | 可统一管理商品、订单、客户和内容 |
| SEO | 每站拥有独立 meta 信息并可生成 sitemap |
| 响应式 | 桌面端与移动端核心流程可正常使用 |

## 一、项目定义

环洗朵官网不是一个单纯承担品牌展示的企业站，而是 iCloush 统一数字平台中**最接近实际商业转化的专业化学品与行业解决方案站点**。在既有平台方法论中，环洗朵承担“专业化学品能力展示 + 行业方案说明 + 咨询线索与采购桥接”的角色，开发期已建议采用 `/tech` 作为站点路由前缀，并与商城、LAB、Care 共用统一账户、商品、订单、内容与 SEO 底座。[1] [2]

根据品牌手册，环洗朵当前具备较完整的业务叙事基础，包括工业洗涤、后厨清洁、住房卫生三大解决方案体系，以及固体洗涤剂、液体洗涤剂、自动分配器、后厨清洁、住房卫生等多类产品矩阵。[3] 因此，本 PRD 的目标不是从零发明一个新站点，而是在统一平台约束下，把这些内容重组为**一条更短、更清晰、更容易留资与申领试样的 B2B 官网链路**。

## 二、产品目标

### 2.1 商业目标

环洗朵官网首要目标是提高专业客户的线索转化效率，而不是追求泛流量曝光。官网需要能够让酒店、餐饮、物业、宿舍管理与相关行业采购人员迅速识别适合自己的解决方案，并完成“咨询、试样、下载 TDS、扫码进入采购入口”中的至少一种动作。

### 2.2 用户目标

对访客而言，官网应在较低理解成本下回答四个问题：第一，环洗朵是否足够专业可信；第二，环洗朵是否理解自己的行业痛点；第三，具体有哪些产品与设备可选；第四，如果现在就想推进合作，下一步该做什么。

### 2.3 系统目标

本项目不重复建设用户、商品、订单、内容系统，而是在既有多站点底座上复用站点路由、后台 CMS、商品中心、内容模块、二维码桥接与后续 JSON API 占位能力。[1] [2]

| 目标层级 | 目标描述 | 对应结果 |
| --- | --- | --- |
| 商业目标 | 提高行业客户咨询、样板申领与采购桥接效率 | 留资量、试样申请量、扫码采购点击量提升 |
| 用户目标 | 降低专业信息理解门槛，缩短从认知到动作的路径 | 更高的页面停留质量与 CTA 转化率 |
| 系统目标 | 复用统一平台底座，减少新站点重复建设成本 | `/tech` 路由复用、后台统一维护、SEO 统一治理 |

## 三、站点定位与品牌任务

环洗朵官网应被定义为：**以专业清洁科技为核心叙事、以解决方案与产品矩阵为主要内容、以样板申领与采购桥接为主转化动作的 B2B 官方站点。**

与 LAB 的角色区分在于，LAB 更适合输出品牌高度与研发气质；环洗朵则需要更直接地进入行业问题、解决方案结构、产品能力、技术参数与商务联系入口。[1] [2]

| 维度 | 环洗朵官网要求 |
| --- | --- |
| 核心定位 | 专业化学品与行业解决方案 B2B 官网 |
| 站点路由 | `/tech` |
| 站点 key / brand code | `tech` / `huanxiduo` |
| 主转化 | 申请试样、咨询方案、下载 TDS、扫码采购 |
| 次转化 | 企业微信加好友、技术总监直连、阅读案例与资质 |
| 内容原则 | 方案优先、参数可读、证据明确、动作直接 |

## 四、目标用户与典型场景

### 4.1 目标用户

| 用户角色 | 主要行业 | 核心诉求 |
| --- | --- | --- |
| 酒店/布草采购负责人 | 酒店、服务式公寓、布草洗涤 | 寻找稳定、低回洗率、可持续的洗涤方案 |
| 后厨/餐饮运营负责人 | 餐饮连锁、中央厨房、食堂 | 解决油污、餐具、地面防滑与清洁效率问题 |
| 住房卫生与物业负责人 | 宿舍、公寓、物业、保洁外包 | 寻找多场景、标准化、成本可控的清洁方案 |
| 渠道商/集采客户 | 行业经销商、集采平台 | 需要产品矩阵、规格、TDS、合作入口 |
| 技术评估者 | 质控、运营、实验室对接人员 | 需要参数、资质、设备方案与可下载文档 |

### 4.2 典型使用场景

访客可能从搜索引擎、微信名片、行业转介绍或品牌物料进入官网首页。他们首先需要被“次时代清洁解决方案”的价值主张吸引，然后快速判断自己的行业归属。进入对应方案页后，访客会进一步查看产品组合、适用机制、实际收益与信任证据。如果已经接近采购阶段，则会进入产品详情页下载 TDS、申请试样，或者扫码跳转至“环洗朵 B2B 采购小程序”完成进一步交易。

## 六、首页设计逻辑

首页的任务不是穷举全部内容，而是用最短路径完成三件事：建立品牌可信度、提供行业分流入口、制造明确动作。

### 6.1 Hero 需求

Hero 采用活性酶分解污垢的 3D 科学可视化视频作为背景，核心标题为**“次时代清洁解决方案”**，并用一句副标题概括其业务能力，例如“面向工业洗涤、后厨清洁与住房卫生的系统性洁净科技方案”。主按钮建议为“申请试样”，次按钮建议为“查看解决方案”。

| 模块 | 内容要求 |
| --- | --- |
| 背景 | 活性酶分解污垢的 3D 模拟或科学可视化视频 |
| 主标题 | 次时代清洁解决方案 |
| 副标题 | 强调三大场景与专业清洁科技能力 |
| 主 CTA | REQUEST SAMPLE / 申请试样 |
| 次 CTA | 查看解决方案 |
| 信任条 | CMA/CNAS、30 年行业经验、三大方案体系等证据摘要 |

### 6.2 首页下游模块建议

| 模块 | 目标 | 内容结构 |
| --- | --- | --- |
| 技术可信模块 | 证明“为什么有效” | 活性酶、自动分配、软水净化、环保承诺、资质 |
| 三大方案入口 | 让访客快速分流 | 工业洗涤 / 后厨清洁 / 住房卫生 |
| 产品矩阵预览 | 承接产品浏览 | 分类卡片 + 代表产品 + 入口按钮 |
| 信任证明模块 | 建立合作安全感 | 资质、客户类型、数据结果、实验室说明 |
| 试样申领模块 | 强制触发主转化 | 简版表单 + 企业微信二维码 + 技术总监联络 |
| 采购桥接模块 | 对接已有交易体系 | 小程序码、JSON API 预留说明、采购指引 |

首页底部页脚必须纳入**企业微信二维码 + 技术总监直连频段**。这里不建议做成传统“联系我们”栏目，而建议定义为“技术联络与采购通道”，让其更贴近 B2B 采购语境。

## 七、解决方案页需求

解决方案模块是环洗朵官网的中段主心骨。它的任务不是讲品牌故事，而是把“行业痛点—方案机制—产品组合—交付结果”组织清楚。

### 7.1 通用页面结构

| 区块 | 说明 |
| --- | --- |
| 场景问题定义 | 说明行业中常见污渍、损耗、效率、环保或成本问题 |
| 环洗朵解决路径 | 说明产品组合、化料逻辑、设备协同与执行流程 |
| 核心产品组合 | 展示适用产品、规格、功能与详情入口 |
| 收益结果 | 强调回洗率、效率、稳定性、环保或标准化改善 |
| CTA 区 | 申请试样、咨询方案、扫码采购 |

### 7.2 三大解决方案页面

品牌手册已清楚给出三大场景，因此官网应完整承接这一结构，而非泛泛而谈。[3]

| 方案页 | 建议重点 |
| --- | --- |
| 工业洗涤 | 布草、制服、强污渍、纤维保护、自动分配、回洗率与综合成本 |
| 后厨清洁 | 油污、餐具、洗碗机、灶台、不锈钢、地面防滑与通渠处理 |
| 住房卫生 | 客房、浴室、地面、织物护理、香氛卫生与易操作性 |

## 八、产品矩阵与 PDP 需求

### 8.1 产品矩阵页

产品矩阵页应支持按类别浏览，而不是只给出长列表。分类至少应覆盖：工业固体洗涤剂、工业液体洗涤剂、自动分配与设备、后厨清洁、住房卫生。[3]

| 需求项 | 说明 |
| --- | --- |
| 分类导航 | 允许按品类与场景切换浏览 |
| 卡片信息 | 产品名、规格、核心功能、适用场景、TDS 状态 |
| 可视化标识 | 用分类徽标与参数标签提升扫读效率 |
| 转化动作 | 查看详情、下载 TDS、申请试样 |

### 8.2 产品详情页（PDP）

PDP 是全站最接近商业动作的页面之一，因此要求比普通企业官网更强的执行导向。除常规产品介绍外，必须在右侧设置**常驻浮窗**，承接 “REQUEST SAMPLE / 申请试样” 主动作，并预留扫码进入采购小程序与企业微信联络入口。

| 区块 | 必需内容 |
| --- | --- |
| 产品基础区 | 产品名称、规格、功能摘要、分类 |
| 场景说明区 | 适用行业、典型污渍/任务、使用建议 |
| 技术参数区 | SKU、规格、适配设备、参数字段、JetBrains Mono 样式数据 |
| 文档区 | TDS 技术规格书下载 |
| 右侧浮窗 | REQUEST SAMPLE、扫码采购、企业微信、技术联络 |
| 延展区 | 推荐搭配产品、相关解决方案、常见问题 |

每个产品详情页必须配置 TDS 下载能力。这不是补充功能，而是环洗朵区别于普通官网的专业门槛。TDS 的内容本身可由后台管理为文件 URL 或统一文档资产字段。

## 九、样板申领主链路

“申领样板”应被视作环洗朵官网的主转化业务，而不仅是一个表单页面。它需要承担线索采集、需求分类、销售分发与后续采购桥接的作用。

### 9.1 页面目标

样板申领页需要让访客在 1 分钟内提交足够有效的信息，并明确下一步会发生什么，例如由技术顾问回访、确认使用场景、匹配产品组合、发出试样或安排进一步沟通。

### 9.2 表单字段建议

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 企业名称 | 是 | B2B 基础识别字段 |
| 联系人 | 是 | 便于跟进 |
| 手机/微信 | 是 | 快速回访渠道 |
| 所属行业 | 是 | 酒店 / 餐饮 / 物业 / 其他 |
| 使用场景 | 是 | 工业洗涤 / 后厨清洁 / 住房卫生 / 多场景 |
| 关注问题 | 否 | 去污力、护色、除味、成本、环保、设备适配等 |
| 预计月用量 | 否 | 便于销售判断样板规格 |
| 地址 | 否 | 样品寄送准备 |
| 备注 | 否 | 补充需求 |

### 9.3 表单提交后的动作

| 动作 | 说明 |
| --- | --- |
| 成功反馈 | 显示“已收到申请，技术顾问将在 1 个工作日内联系” |
| 企业微信引导 | 提供企业微信二维码，允许访客同步添加 |
| 小程序桥接 | 提供采购小程序码作为补充动作 |
| 后台沉淀 | 写入统一线索库，按品牌与场景归属到环洗朵 |
| 通知机制 | 可调用现有 owner/运营通知能力提示新线索到达 |

## 十、转化路径设计

环洗朵官网的主链路应围绕“看懂—信任—动作”三步展开，而不是让用户在大量页面中自行摸索。

| 路径编号 | 起点 | 中间环节 | 终点 |
| --- | --- | --- | --- |
| Path A | 首页 Hero | 解决方案页 | 申请试样 |
| Path B | 首页/方案页 | 产品矩阵 | 产品详情页 + TDS 下载 |
| Path C | 产品详情页 | 右侧浮窗 | 扫码进入 B2B 采购小程序 |
| Path D | 任意页面底部 | 企业微信二维码 / 技术总监直连 | 商务咨询 |
| Path E | 搜索进入产品页 | 查看参数与 TDS | 留资或采购桥接 |

为了避免用户迷路，建议在全站固定三类高频操作入口：顶部导航中的“申领样板”、产品页右侧浮窗中的“REQUEST SAMPLE”、以及页脚中的“技术联络与采购通道”。

## 十一、技术实现边界

本次官网 PRD 明确要求复用既有统一平台底座，不另起一套独立用户、订单与后台系统。[1] [2]

### 11.1 路由与站点配置

| 项目 | 约束 |
| --- | --- |
| 站点路由 | 复用 `/tech` 前缀 |
| 品牌归属 | `huanxiduo` |
| 站点 key | `tech` |
| 页面实现 | 优先复用现有 web-b2b 多站点结构或同等前台框架 |
| 内容治理 | 复用 Admin 已有 site config / solutions / cases / logos 等内容模型 |

### 11.2 数据与接口边界

官网不需要新建完整商城能力，但需要与既有平台能力形成桥接。建议采用以下原则。

| 能力 | 建议实现 |
| --- | --- |
| 产品数据 | 复用已有产品中心与品牌归属数据 |
| 内容模块 | 复用 Admin 内容管理能力 |
| 线索表单 | 复用统一 leads 或新增品牌场景字段 |
| TDS 文件 | 作为产品附属文档字段存储在统一内容资产体系 |
| 扫码采购 | 复用现有二维码/外链配置逻辑，指向“环洗朵 B2B 采购小程序” |
| 支付/下单 | 保留 JSON API 桥接，不在官网内直接重造支付前端 |

### 11.3 JSON API 预留建议

用户已明确要求预留 JSON API，并通过扫码进入 B2B 采购小程序。基于这一目标，可在 PRD 层先定义三类接口方向。

| 接口方向 | 用途 |
| --- | --- |
| 样板申请接口 | 接收试样申请表单并回写线索库 |
| 采购入口配置接口 | 返回小程序路径、二维码 URL、外链说明等桥接信息 |
| 产品文档接口 | 返回产品对应的 TDS/规格文件元数据 |

## 十二、后台内容管理要求

为了保证官网不是“一次性静态页面”，后台必须支持围绕环洗朵站点进行细粒度内容配置。

| 内容对象 | 后台要求 |
| --- | --- |
| 首页 Hero | 可配置标题、副标题、视频或封面资源 |
| 解决方案模块 | 可配置方案标题、问题、产品组合、收益描述 |
| 产品详情 | 可配置卖点、参数、适用场景、TDS 文件、相关产品 |
| 联系通道 | 可配置企业微信二维码、技术总监联络描述、直连频段 |
| 采购桥接 | 可配置小程序二维码、路径、外链 |
| 信任背书 | 可配置资质、案例、客户 Logo 墙 |

这类管理能力与当前 Admin 已经承载的多站点内容治理方向一致，因此 PRD 层面应明确：**环洗朵官网必须是一个可运营的品牌站，而不是前端硬编码专题页。**

## 十三、SEO 与内容治理需求

环洗朵站点在平台 PRD 中已被纳入独立站点 SEO 治理范围，因此本次官网设计也必须保持站点级 SEO 独立性。[2]

| SEO 项 | 要求 |
| --- | --- |
| 页面标题 | 首页、方案页、产品页、案例页均需独立 title |
| 页面描述 | 与场景和产品内容强相关，避免通用公司介绍式描述 |
| URL 结构 | 使用清晰的 `/tech/...` 分层 slug |
| TDS 页面/文件 | 文件名与产品 slug 对齐，便于外部引用 |
| 站内链接 | 首页 → 方案 → 产品 → TDS/试样 形成清晰链路 |
| Sitemap | 继续纳入统一 sitemap 机制 |

内容治理层面，建议坚持一个原则：**每一页都必须服务于“解决一个行业问题”或“推进一个明确动作”。** 对于无助于转化的空泛宣传段落，应尽量减少。

## 十四、响应式体验需求

环洗朵官网的移动端不是桌面端的裁剪版本，而是采购场景下的高频入口，因此移动端必须重点保证 CTA、二维码、表单与文档下载可用性。

| 终端 | 设计要求 |
| --- | --- |
| 桌面端 | 强调技术可信、方案全貌与参数展示效率 |
| 平板端 | 保持方案页图文层级清晰，避免卡片过密 |
| 移动端 | 固定主 CTA、表单易填、二维码清晰、TDS 下载可触达 |

特别需要注意的是，PDP 右侧常驻浮窗在移动端应转为底部悬浮 CTA 条或抽屉式操作区，以确保“申请试样”与“扫码采购”始终可见。

## 十五、验收标准

| 模块 | 验收标准 |
| --- | --- |
| 首页 | Hero 视频、主文案、三大解决方案入口与主 CTA 完整可用 |
| 信息架构 | 四大导航“洁净科技 / 解决方案 / 产品矩阵 / 申领样板”完整建立 |
| 解决方案 | 三大解决方案页结构清晰，具备产品组合与 CTA |
| 产品体系 | 产品矩阵页与 PDP 可浏览，PDP 含 TDS 下载与浮窗 CTA |
| 转化链路 | 表单、企业微信、扫码采购至少三种动作可达 |
| 平台集成 | 复用 `/tech` 路由、后台内容配置与统一线索沉淀能力 |
| SEO | 页面具备独立 title/description/slug，并纳入 sitemap |
| 响应式 | 桌面端与移动端关键转化动作可顺畅完成 |

## 十六、建议实施顺序

如果用户确认本 PRD，建议按以下顺序推进前端实现。首先完成首页、解决方案总页、产品矩阵页与 PDP 的视觉和结构落地，因为这是最直接影响品牌感知与转化的部分。其次完成样板申领页与后台内容字段补齐，确保 CTA 不是空动作。再次完善 TDS 文档管理、二维码采购桥接与企业微信联络区。最后再补充案例页、资质页和 SEO 精修。

## 十七、结论

环洗朵官网的正确方向，不是做成又一个标准企业官网，也不是复制 LAB 的高张力黑域视觉，而是建设一个**基于统一平台底座、以行业方案和产品能力为主、以试样申领和采购桥接为核心动作的 B2B 官方站点**。[1] [2] [3]

只要这一点成立，官网的所有设计判断都会变得清晰：导航必须围绕业务动作，首页必须先讲能力与场景，产品页必须有 TDS 与样板入口，页脚必须承担联络与采购通道，后台必须支持持续运营，而技术实现则应最大化复用现有 `/tech` 站点与统一内容/商品/线索体系。

# iCloush 支付 API 清单与分期接入建议

作者：**Manus AI**  
日期：2026-04-10

当前阶段我已按你的要求，**暂时跳过微信与支付宝 JSAPI 的真实 API 接入**，后续开发优先继续推进订单查询、线下转账审核、客户中心与后台运营能力。为了保证后续真正接入时不返工，这里先把微信、支付宝以及配套能力所需的 API 全部梳理成实施清单。

从支付闭环角度看，**真正可上线的支付集成绝不只是一条“下单接口”**。无论是微信还是支付宝，至少都要同时具备“创建交易、异步通知、主动查单、退款、退款结果确认、签名验签与证书材料管理”这几类能力，否则系统无法在回调丢失、用户中断、退款售后或风控排障时保持订单状态一致。[1] [2] [3] [4]

## 一、当前建议的开发边界

本轮先不接入真实支付网关，而是把支付相关工作拆成两个层次。第一层是**业务闭环层**，也就是订单创建、支付方式选择、线下转账凭证上传、后台审核、订单查询、财务跟踪和状态流转，这些都可以先做完。第二层才是**真实渠道层**，即与微信、支付宝进行联调、签名、证书配置、沙箱或生产验证。

| 层级 | 当前是否开发 | 说明 |
| --- | --- | --- |
| 订单与支付业务模型 | 是 | 继续开发，包含订单、支付单、凭证、审核、状态流转 |
| 线下转账审核闭环 | 是 | 继续开发，优先满足 B2B 对公转账场景 |
| 微信 JSAPI 真实下单 | 否 | 暂缓，只保留草稿能力与接口占位 |
| 支付宝真实线上支付 | 否 | 暂缓，只先完成 API 清单与后续接入设计 |
| 退款与转账真实调用 | 否 | 暂缓，但需提前保留数据结构与状态位 |

## 二、微信支付所需 API 清单

微信支付 JSAPI 官方文档明确说明，JSAPI/小程序支付的核心下单接口为 `POST /v3/pay/transactions/jsapi`，并且配套存在调起支付、查单、关单、支付成功回调、退款申请、退款查询、退款回调等接口族。[1] [5]

| 渠道 | 能力 | 是否上线必需 | 方法 | 接口/能力标识 | 用途 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| 微信支付 | JSAPI 下单 | 是 | POST | `/v3/pay/transactions/jsapi` | 生成 `prepay_id`，供公众号/小程序调起支付 | 真实收银的入口接口 [1] |
| 微信支付 | JSAPI 调起支付参数生成 | 是 | 本地签名 | 前端支付参数二次签名 | 将 `appId`、`timeStamp`、`nonceStr`、`package`、`signType`、`paySign` 返回前端 | 不是远程 API，但属于必须实现的服务端能力 [5] |
| 微信支付 | 支付结果通知 | 是 | POST | 商户自定义 `notify_url` | 异步确认支付成功、金额、商户单号、微信单号 | 需要验签与解密 [1] |
| 微信支付 | 商户单号查单 | 是 | GET | `/v3/pay/transactions/out-trade-no/{out_trade_no}` | 回调丢失时主动核单 | 补偿任务必需 [5] |
| 微信支付 | 微信支付单号查单 | 否，但建议 | GET | `/v3/pay/transactions/id/{transaction_id}` | 通过微信侧单号追踪交易 | 便于人工排障 |
| 微信支付 | 关闭订单 | 否，但强烈建议 | POST | `/v3/pay/transactions/out-trade-no/{out_trade_no}/close` | 未支付超时订单关闭 | 避免重复支付与脏单 [1] |
| 微信支付 | 退款申请 | 二期建议必做 | POST | `/v3/refund/domestic/refunds` | 售后退款 | 一旦涉及售后就应尽快接入 [2] |
| 微信支付 | 退款查询 | 二期建议必做 | GET | 按商户退款单号查询单笔退款 | 退款补偿与人工核对 | 退款状态可能延迟 [6] |
| 微信支付 | 退款结果通知 | 二期建议必做 | POST | 商户自定义 `refund_notify_url` | 异步更新退款状态 | 与退款申请配套 [2] |
| 微信支付 | 商家转账 | 视业务决定 | POST | `/v3/transfer/batches` | 返佣、渠道结算、返利等主动打款 | 非收款闭环必需 [7] |

## 三、支付宝所需 API 清单

支付宝统一收单侧，当前最小可用闭环至少包括**创建交易、异步通知、主动查单、退款、退款结果确认**。支付宝官方文档将 `alipay.trade.query` 作为统一收单交易查询接口，用于商户主动查询订单状态并完成后续业务逻辑；退款接口文档则明确指出，退款调用成功后仍需要结合返回字段或后续查询确认最终退款结果。[3] [4]

| 渠道 | 能力 | 是否上线必需 | 方法 | 接口/能力标识 | 用途 | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| 支付宝 | 创建交易 | 是 | POST | `alipay.trade.create` | 创建交易单 | 不同终端形态要区分 `product_code`、`buyer_id`/`open_id` |
| 支付宝 | 前端拉起收银台/JSAPI | 是 | 本地拼装请求 | 由服务端签名生成前端拉起参数 | 视 H5、生活号、JSAPI 或小程序形态而定 | 不是独立网关接口，但属于必做能力 |
| 支付宝 | 支付结果通知 | 是 | POST | 商户自定义 `notify_url` | 支付成功异步通知 | 需验签 [3] |
| 支付宝 | 交易查询 | 是 | POST | `alipay.trade.query` | 回调缺失、超时或对账时主动确认交易结果 | 最小闭环必需 [3] |
| 支付宝 | 关闭交易 | 否，但建议 | POST | `alipay.trade.close` | 未支付交易关闭 | 建议与超时订单清理一起做 |
| 支付宝 | 退款申请 | 二期建议必做 | POST | `alipay.trade.refund` | 售后退款 | 需结合退款结果判断 [4] |
| 支付宝 | 退款查询 | 二期建议必做 | POST | `alipay.trade.fastpay.refund.query` | 查询退款最终状态 | 退款排障必需 |
| 支付宝 | 单笔转账 | 视业务决定 | POST | `alipay.fund.trans.uni.transfer` | 佣金、返利、商家打款 | 非收款闭环必需 |

## 四、除了 API 本身，还必须同步准备的能力

真正接入线上支付时，工程工作量大头往往不是“调用 API”本身，而是**签名、回调、幂等、状态机、证书和对账**。如果这些配套能力没有先设计好，即便支付接口能调通，也很难稳定上线。[1] [3]

| 类别 | 微信支付需要 | 支付宝需要 | 对 iCloush 的含义 |
| --- | --- | --- | --- |
| 商户身份材料 | 商户号、AppID、API v3 Key、商户私钥、平台证书序列号 | `app_id`、应用私钥、支付宝公钥、网关地址 | 必须通过密钥管理配置，不应硬编码 |
| 用户身份 | `openid` | `buyer_id` 或 `open_id` | 需要在会员/登录体系中留出映射字段 |
| 回调处理 | 支付回调、退款回调、验签、解密 | 支付回调、退款结果确认、验签 | 必须建立幂等更新机制 |
| 主动补偿 | 查单、退款查询 | 查单、退款查询 | 需要定时任务或手工补偿入口 |
| 状态机 | 待支付、支付中、支付成功、关闭、退款中、已退款 | 同左 | 必须与订单状态分离但可联动 |
| 对账 | 下载账单/对账能力 | 账务核对能力 | 财务上线前建议预留 |
| 风控与异常 | 金额校验、重复通知处理、签名失败告警 | 签名失败、金额不一致、回调重试 | 必须记录审计日志 |

## 五、按你当前要求，建议的接入顺序

既然你要求**先开发其他部分**，我建议把真实支付渠道接入放在业务主干跑通之后再做。顺序上应先让平台具备“可下单、可查询、可线下付款、可后台审核、可追踪”的业务能力，再去接线上渠道，这样联调时只需要替换支付通道层，不需要重做订单域模型。

| 阶段 | 建议优先级 | 目标 |
| --- | --- | --- |
| 第一阶段 | 最高 | 完成订单中心、支付记录、线下转账凭证上传、后台审核、客户查询 |
| 第二阶段 | 高 | 接入微信 JSAPI：下单、调起、回调、查单 |
| 第三阶段 | 高 | 接入支付宝：创建交易、拉起支付、回调、查单 |
| 第四阶段 | 中 | 接入退款：微信退款、支付宝退款、退款查询、退款回调/确认 |
| 第五阶段 | 视业务决定 | 接入商家转账、返佣、结算、自动对账 |

## 六、面向当前项目的最小字段准备建议

为了避免后续接入微信、支付宝时再改数据库，当前订单与支付模型建议至少预留如下信息。当前项目里已经存在订单、支付单和线下转账凭证结构，因此后续主要是补充线上渠道特有字段与签名材料映射。

| 对象 | 建议字段 | 作用 |
| --- | --- | --- |
| 订单 | `orderNo`、`status`、`payableAmount`、`currency` | 业务主单 |
| 支付单 | `paymentNo`、`provider`、`paymentScenario`、`status`、`providerTradeNo`、`metaJson` | 支付渠道实例 |
| 用户/会员 | `wechatOpenId`、`alipayOpenId` 或等效映射 | 发起 JSAPI / 支付宝端内支付所需 |
| 回调日志 | 原始请求体、验签结果、处理结果、重试次数 | 审计与排障 |
| 退款单 | `refundNo`、`paymentId`、`amount`、`status`、`providerRefundNo` | 售后闭环 |

## 七、结论

结论上，若以“能够正式上线支付”为标准，**微信与支付宝都不是只接一条 API 就够**。微信最小闭环应至少覆盖 `JSAPI下单 + 调起签名 + 支付回调 + 主动查单`，支付宝最小闭环应至少覆盖 `交易创建 + 前端拉起参数生成 + 支付回调 + 主动查单`；如果平台要支持售后，则两边都应同步规划退款及退款确认能力。[1] [2] [3] [4] [5] [6]

按你现在的要求，我会把**真实支付接入继续后置**，先优先开发其他模块，并将当前项目中的支付部分保留为“可扩展占位 + 线下审核闭环 + 线上渠道清单已明确”的状态。等你确认开始接微信或支付宝时，我就可以直接按这份清单逐项落地。

# iCloush 支付接入 API 清单（暂缓 JSAPI 真实接入版）

## 背景说明

当前阶段已按照你的要求，**先跳过微信支付与支付宝 JSAPI 的真实联调**，优先推进平台其他模块开发。但为了避免后续支付接入返工，系统已经先补齐支付身份映射、支付回调日志与退款单等底层数据结构，并整理出正式接入时需要实现的 API 清单、回调链路与关键参数约束。

这份清单的作用不是立即接入渠道，而是作为下一阶段支付开发的**接口范围基线**。也就是说，后续只要开始做真实支付，就应以本文档中的接口集合、回调处理、验签要求和内部服务封装作为最小闭环。

## 一、微信支付需要的 API 清单

微信支付 JSAPI 的官方开发链路明确由“下单获取 `prepay_id`、前端调起支付、支付后查单确认、支付成功回调、退款申请、退款查询、账单下载”组成，其中 `openid`、`time_expire`、`prepay_id` 和 `JSAPI 支付授权目录` 都属于接入时必须处理的关键要素。[1][2]

| 分类 | 官方能力 | 典型接口/对象 | 用途 | 是否首期必须 |
| --- | --- | --- | --- | --- |
| 支付准备 | 获取用户 `openid` | 微信 OAuth / 公众号或小程序侧用户标识获取流程 | JSAPI 下单必须带 `openid`，用于标识付款用户 | 是 |
| 核心支付 | JSAPI 下单 | `POST /v3/pay/transactions/jsapi` | 创建支付单并返回 `prepay_id` | 是 |
| 前端拉起 | 微信内调起收银台 | `WeixinJSBridge` | 在微信内网页拉起支付控件 | 是 |
| 支付确认 | 查询订单 | 按微信支付订单号或商户订单号查单 | 前端返回成功、取消或异常后，服务端二次确认订单状态 | 是 |
| 生命周期控制 | 关闭订单 | 关闭未支付订单 | 超时未支付或业务取消时关单 | 是 |
| 回调 | 支付成功回调通知 | 支付结果通知接口 | 服务端异步确认支付成功并落库 | 是 |
| 退款 | 申请退款 | `POST /v3/refund/domestic/refunds` | 全额或部分退款 | 是 |
| 退款确认 | 查询退款单 | 按商户退款单号/微信退款单号查询 | 确认退款结果，避免仅凭受理状态变更业务单 | 是 |
| 退款回调 | 退款结果回调通知 | 退款结果通知接口 | 异步接收退款完成或异常结果 | 是 |
| 对账 | 申请/下载交易账单 | 交易账单相关接口 | 财务对账与异常核查 | 次期 |
| 对账 | 申请/下载资金账单 | 资金账单相关接口 | 资金流水核对 | 次期 |
| 安全 | 平台证书/微信支付公钥 | 平台证书、公钥下载与轮换 | 验签、解密回调报文、校验平台身份 | 是 |

### 微信支付首期关键参数

| 场景 | 关键字段 | 说明 |
| --- | --- | --- |
| 下单 | `appid`、`mchid`、`description`、`out_trade_no`、`notify_url`、`amount.total`、`payer.openid` | 其中 `payer.openid` 是 JSAPI 必填，`notify_url` 用于服务端接收支付结果 |
| 超时控制 | `time_expire` | 可限制用户支付最晚时间，超时后需配合关单处理 [1] |
| 调起支付 | `prepay_id`、时间戳、随机串、签名包 | `prepay_id` 有效期为 2 小时，过期需重新下单 [1] |
| 退款 | `out_refund_no`、`reason`、`notify_url`、`amount.refund`、`amount.total`、`currency` | 单笔订单最多支持 50 次部分退款，需结合退款通知与查询确认最终状态 [2] |

## 二、支付宝需要的 API 清单

支付宝手机网站支付的接入流程与微信类似，也必须形成“下单、同步返回、异步通知、主动查单、关闭未支付订单、退款、退款查询、签名验签”的闭环。根据支付宝官方文档与官方支持中心检索结果，首期应至少覆盖 `alipay.trade.wap.pay`、`alipay.trade.query`、`alipay.trade.close`、`alipay.trade.refund`、`alipay.trade.fastpay.refund.query`，并配套 `notify_url` 异步通知和 `RSA2` 验签。[3][4][5]

| 分类 | 官方能力 | 典型接口/对象 | 用途 | 是否首期必须 |
| --- | --- | --- | --- | --- |
| 核心支付 | 手机网站支付下单 | `alipay.trade.wap.pay` | 创建支付宝支付请求并跳转收银台 | 是 |
| 支付确认 | 统一收单交易查询 | `alipay.trade.query` | 用户支付返回后，服务端主动确认交易状态 | 是 |
| 生命周期控制 | 统一收单交易关闭 | `alipay.trade.close` | 关闭未支付或不再继续支付的交易 | 是 |
| 回调 | 异步通知 | `notify_url` | 接收支付结果通知，避免只依赖前端跳转结果 | 是 |
| 安全 | 签名验签 | `RSA2` 验签、公钥配置 | 校验回调与接口报文真实性 | 是 |
| 退款 | 统一收单交易退款 | `alipay.trade.refund` | 发起全额或部分退款 | 是 |
| 退款确认 | 统一收单交易退款查询 | `alipay.trade.fastpay.refund.query` | 核对退款是否成功、处理中或失败 | 是 |
| 对账 | 对账单下载 | 账单查询/下载相关能力 | 财务核账与差错处理 | 次期 |

### 支付宝首期关键参数

| 场景 | 关键字段 | 说明 |
| --- | --- | --- |
| 下单 | `out_trade_no`、`total_amount`、`subject`、`product_code`、`quit_url`、`return_url`、`notify_url` | `notify_url` 是服务端异步确认核心；`return_url` 只负责用户跳转展示，不能作为支付成功依据 |
| 查单 | `out_trade_no` 或 `trade_no` | 建议同时保存平台单号与商户单号，便于追踪与对账 |
| 关单 | `out_trade_no` 或 `trade_no` | 仅用于未完成支付的交易收口 |
| 退款 | `out_request_no`、`refund_amount`、`refund_reason` | 建议每次退款都生成独立退款请求号，便于部分退款追踪 |
| 通知验签 | `notify_id`、`sign`、`sign_type`、交易状态字段 | 必须使用支付宝公钥按 `RSA2` 做验签后，才能更新本地支付状态 [4][5] |

## 三、平台内部还需要预留的服务 API

即使暂缓真实渠道接入，平台内部仍需要一组**自有服务接口**作为支付编排层，否则后续前端会直接依赖第三方格式，导致品牌维度、订单维度和日志维度难以统一。

| 内部接口 | 建议用途 | 说明 |
| --- | --- | --- |
| `POST /api/payments/:orderNo/create-wechat-jsapi` | 创建微信支付请求 | 负责组装微信下单参数、落库支付意图、返回前端调起参数 |
| `POST /api/payments/:orderNo/create-alipay-wap` | 创建支付宝支付请求 | 负责组装支付宝下单参数、落库支付意图、返回跳转表单或链接 |
| `POST /api/payments/wechat/notify` | 微信支付回调入口 | 验签、解密、写入支付日志、推进订单状态 |
| `POST /api/payments/alipay/notify` | 支付宝回调入口 | 验签、写入支付日志、推进订单状态 |
| `GET /api/payments/:orderNo/status` | 聚合支付状态查询 | 先查本地状态，必要时补查第三方渠道 |
| `POST /api/refunds/:orderNo/apply` | 统一退款申请 | 根据订单渠道路由到微信或支付宝退款接口 |
| `GET /api/refunds/:refundNo/status` | 聚合退款状态查询 | 拉齐本地退款单与渠道退款状态 |
| `POST /api/payments/:orderNo/close` | 聚合关单 | 超时订单或取消订单统一关闭 |
| `POST /api/payments/reconcile/run` | 对账任务触发 | 触发账单下载、比对与异常标记 |

## 四、必须实现的异步回调与状态机

真实支付接入时，**前端返回不能直接当作支付成功依据**。微信官方开发指引明确要求在用户从支付界面回到商户页面后，由服务端调用查单接口再次确认状态；同时微信还会发送支付成功回调。退款也是同样逻辑，退款申请成功只代表受理成功，最终结果仍应依赖退款回调与退款查询。[1][2]

支付宝同理，`return_url` 只负责页面跳转，真正的交易确认必须依赖 `notify_url` 异步通知与服务端查单。也就是说，平台内部状态机至少要覆盖以下几类状态：**待支付、支付处理中、支付成功、支付失败、已关闭、退款处理中、退款成功、退款失败**。此外，还需要保存所有原始通知载荷、验签结果、幂等处理结果与重试记录，避免出现重复通知造成的状态回退。

## 五、当前阶段建议的开发顺序

既然当前先不做真实 JSAPI 联调，那么后续最合理的顺序应是：先继续完成平台非支付功能；并行完善支付编排层的数据结构、日志表和内部接口契约；最后在拿到微信商户号、商户证书、支付宝应用、公钥私钥和回调域名后，再进入真实联调。这样可以把当前开发与后续收单渠道接入解耦，降低对业务页面推进的阻塞。

| 顺序 | 开发项 | 当前建议 |
| --- | --- | --- |
| 1 | 继续开发非支付业务模块 | 立即继续 |
| 2 | 完善支付编排层内部接口与 DTO | 可并行进行 |
| 3 | 完成回调日志、退款单、支付身份映射落库逻辑 | 可并行进行 |
| 4 | 配置微信支付授权目录、商户证书、公钥 | 等真实接入时再做 |
| 5 | 配置支付宝应用、公钥私钥、异步通知地址 | 等真实接入时再做 |
| 6 | 执行沙箱或正式环境联调 | 放到下一阶段 |

## 六、当前已完成与待完成

当前已经完成的部分，是**支付接入前的数据底座准备**，包括支付身份映射、支付回调日志与退款单相关结构。当前尚未完成的部分，是**第三方真实接口接入、签名验签、前端拉起支付、异步回调处理和对账任务**。因此，接下来完全可以按照你的要求，先继续开发其他业务模块，而不被微信或支付宝联调阻塞。

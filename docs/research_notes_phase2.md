# 阶段二研究笔记

## B2B 商城建设关键原则

基于 BigCommerce 关于 B2B 电商网站建设的公开文章，B2B 商城不是传统零售站点的简单复制，而是需要围绕企业采购流程进行设计。其核心特征包括清晰的业务目标设定、可管理的预算与实施节奏、面向企业采购场景的功能设计，以及与现有 ERP、CRM、库存系统的集成能力。[1]

对本项目而言，这意味着统一平台中的商城系统应优先满足 **企业客户采购效率**，而不是以单纯营销展示为中心。商城前台需要具备产品展示、分类导航、搜索与筛选、购物车、订单历史、批量采购和快捷复购等典型能力；后台则需要支持客户分层、价格策略、订单协同处理和内容统一分发。[1]

## 对 iCloush 统一平台的直接启示

第一，平台架构上应采用“统一底座 + 多站点前台”的设计思路，即共享用户、商品、订单、内容与 SEO 能力，但在品牌层面通过独立站点入口、独立内容结构与独立视觉主题来体现 iCloush LAB.、环洗朵科技和富朵朵 iCloush Care 的差异化定位。BigCommerce 文章强调了 multi-storefront 与 customer groups 等能力的重要性，这与本项目的多站点目标高度一致。[1]

第二，B2B 交易流程必须兼顾企业客户的复杂采购需求。文章指出，企业客户通常需要账户管理、自定义价格、快速复购、发票处理和支付方式控制等功能。[1] 这意味着本项目即使在第一阶段先建设标准购物车和订单系统，也应在 PRD 中预留企业客户分级、协议价、询价单、审批流与线下对账等扩展位。

第三，在实施方法上，项目应先完成业务目标、站点定位、信息架构和系统边界的统一定义，再进入详细交互和工程实施阶段。否则多品牌站点与商城、后台并行推进时，很容易出现路由冲突、数据结构重复和品牌表达混乱的问题。[1]

## Stripe 支付集成对本项目的启示

根据 Stripe 官方文档，Checkout 是一种低代码支付集成方式，支持托管支付页、嵌入式支付页和可定制元素式支付流程，并可覆盖一次性付款与订阅场景。[2] 对当前项目而言，商城系统第一阶段最适合采用 **Stripe Checkout Session + 服务端订单创建 + webhook 异步确认履约** 的方式，这样可以在保证安全性与开发效率的同时，减少自行处理支付表单、PCI 合规和多支付方式兼容的复杂度。[2]

Stripe 文档明确指出，标准支付生命周期应包含四个关键步骤：应用创建 Checkout Session、将用户重定向到支付页、用户完成支付、服务端通过 `checkout.session.completed` 事件执行订单履约。[2] 这对于本项目的订单系统设计非常关键，因为它意味着订单状态不能只依赖前端跳转结果，而应由后端 webhook 作为支付成功的准确信号源。

另外，Stripe Checkout 内置响应式移动端设计、PCI 合规、卡片校验、错误提示和多种支付方式支持。[2] 这意味着在平台首期建设中，可以把更多研发资源放在商城、账户、订单和多站点内容管理能力上，而不是优先自研复杂支付页。

## SEO 与 sitemap 对多站点平台的启示

根据 Google Search Central，sitemap 是向搜索引擎说明网站页面、文件及其关系的结构化文件，可帮助搜索引擎更高效抓取站点内容，尤其适用于较大、较新或结构复杂的网站。[3] 由于本项目包含一个商城系统与三个独立品牌官网，且每个站点都存在独立内容树和独立入口，因此应将 sitemap 作为基础能力纳入平台架构，而不是上线后的补充优化项。[3]

对统一平台而言，更合理的做法不是只生成一个单一 sitemap，而是为商城、iCloush LAB.、环洗朵科技、富朵朵 iCloush Care 各自生成独立 sitemap，并通过 sitemap index 进行统一管理。这样既便于后续站点级 SEO 运营，也更符合多站点内容治理、分站收录监控和独立品牌扩展的需要。[3]

Google 同时强调，站点内部清晰的链接结构仍然是抓取和收录的前提，sitemap 不能替代良好的信息架构。[3] 因此本项目在 PRD 中需要把“站点导航、内容层级、站内跳转逻辑、产品与解决方案互链”与 sitemap 视作同等重要的 SEO 基础设施。

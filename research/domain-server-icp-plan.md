## 二、域名资源的主次策略建议

从当前产品结构看，iCloush 并不是单一官网，而是一个包含**统一入口、商城、客户中心、后台管理、品牌子站、接口服务**的多站点平台。对于这类体系，第一阶段最稳妥的策略不是让每个品牌立即独立使用一个顶级域名，而是以一个**主品牌主域名**承接备案、证书、解析和 SEO 主入口，再将其他业务收敛到子域名与必要的品牌保护域名上。

结合您现有的域名资源，当前更适合采用 **`icloush.com` 作为主生产域名** 的方案。因为腾讯云与主流备案规则均强调，备案是围绕**主域名**及其主体信息展开；主域备案完成后，其下子域名可以在同一备案框架中承载多个业务入口，从执行效率和运维成本上都更优。[1] [3]

| 域名 | 当前定位建议 | 第一阶段是否承载正式业务 | 建议角色 |
| --- | --- | --- | --- |
| `icloush.com` | 集团总入口 / 主品牌域名 | 是 | **主备案域名、主站入口、证书主域** |
| `shop.icloush.com` | 商城前台 | 是 | 商品展示、下单与客户访问入口 |
| `account.icloush.com` | 客户中心 | 是 | 登录、订单、工单、账户信息 |
| `admin.icloush.com` | 管理后台 | 是 | 运营后台，建议限权、加 WAF/MFA |
| `api.icloush.com` | API 网关 | 是 | 统一后端接口入口 |
| `lab.icloush.com` | LAB 品牌站 | 建议是 | 研发/实验室品牌站点 |
| `tech.icloush.com` | 环洗朵科技站 | 建议是 | 行业解决方案与企业介绍 |
| `care.icloush.com` | Care 服务站 | 建议是 | 奢护服务与咨询入口 |
| `icloushlab.com` / `icloushlab.cn` | 已注册品牌域 | 否 | 品牌保护、301 跳转或后续独立站储备 |
| `huanxiduo.cn` | 已注册品牌域 | 视品牌策略决定 | 可作为第二阶段独立域，首期建议先保留 |
| `icloushcare.com` | 待注册 | 否 | 品牌保护域名，优先注册但不建议首期单独承载业务 |

这套方案的核心思想是：**先统一、后拆分**。在备案、解析、部署、证书与内容运营体系未完全稳定之前，以 `icloush.com` 为母域统一承载，是风险最低且最利于快速上线的路径。等后续 LAB、环洗朵科技或 Care 形成独立营销与独立投放需求后，再考虑将对应品牌切换为独立顶级域名站点。

## 三、云服务器后端部署架构的推荐形态

当前项目不是一个纯静态站点，而是围绕 monorepo 的多应用结构展开，至少包含前台业务入口、后台管理与接口服务。因此，即便第一阶段采用腾讯云轻量应用服务器，也应当把架构设计成“**一台应用服务器承载接入层与应用层 + 独立数据库/对象存储**”的轻生产结构，而不是把数据库、文件和所有运行时全部强耦合在同一台轻量实例上长期运行。

| 架构层级 | 推荐组件 | 作用 | 第一阶段建议 |
| --- | --- | --- | --- |
| 边缘接入层 | DNSPod / 腾讯云解析 + HTTPS 证书 + Nginx | 域名接入、TLS 终止、反向代理、路由分发 | 必须配置 |
| 应用层 | Node.js 运行时 + PM2 / systemd | 运行前台站点、后台与 API 服务 | 必须配置 |
| 数据层 | 托管 MySQL / TiDB | 用户、订单、商品、内容、线索等业务数据 | **不建议长期与应用混部** |
| 文件层 | COS / S3 兼容对象存储 | 图片、附件、资料、媒体文件 | 必须配置 |
| 安全层 | WAF、基础防火墙、安全组、登录审计 | 降低后台暴露风险 | 建议同步配置 |
| 运维层 | 日志、备份、监控与告警 | 发布、回滚、故障排查 | 必须配置 |

如果继续沿着您已经选定的腾讯云方案落地，那么第一阶段的最小生产架构建议如下：

| 组件 | 当前建议 | 原因 |
| --- | --- | --- |
| 计算资源 | 腾讯云轻量应用服务器 2 核 4G | 对首期品牌站、客户中心、后台和轻量 API 足够起步 |
| 地域 | 上海 | 与您当前购买页一致，延续性最好 |
| 系统 | Ubuntu 24.04 LTS | 便于 Node、Nginx、PM2、Docker 生态部署 |
| 数据库 | 托管数据库优先 | 避免数据库与应用抢占单机资源 |
| 对象存储 | 腾讯云 COS 或现有 S3 兼容存储 | 统一图片与媒体文件管理 |
| 守护方式 | PM2 或 systemd | 简化 Node 服务守护与重启 |
| 反向代理 | Nginx | 最适合多子域与多服务统一入口分发 |

需要强调的是，**2 核 4G 是一个合理的第一阶段起步配置，但更像“可上线、可验证、可承载中小流量”的初始生产规格，而不是长期高并发终局架构**。如果后续商城查询、后台管理、图片处理、定时任务与接口并发明显增大，可以再平滑升级到 4 核 8G 或拆分为“前台/后台/API”分层部署。

## 四、推荐的子域名与服务映射表

为了避免上线前反复改域名和反向代理规则，建议现在就把目标域名与服务关系固定下来。只要这张映射表确定，后续 DNS 解析、Nginx server block、SSL 证书申请与备案材料填写都会变得非常顺。

| 子域名 / 域名 | 对应系统 | 建议接入方式 | 备注 |
| --- | --- | --- | --- |
| `icloush.com` | 统一平台首页 / 集团入口 | Nginx 站点主入口 | 对外品牌主首页 |
| `www.icloush.com` | 主站别名 | 301 跳转到主站 | 统一搜索权重 |
| `shop.icloush.com` | 商城前台 | 独立虚拟主机或独立进程反代 | 与交易场景分离 |
| `account.icloush.com` | 客户中心 | 独立路由或独立前端进程 | 对登录态要求更高 |
| `admin.icloush.com` | 管理后台 | 独立后台进程 + 限制访问策略 | 强烈建议加白名单/MFA |
| `api.icloush.com` | 后端 API | 反向代理至 API 服务 | 统一接口域名 |
| `lab.icloush.com` | LAB 品牌站 | 静态/SSR 页面 | 与内容营销挂钩 |
| `tech.icloush.com` | 环洗朵科技站 | 静态/SSR 页面 | 首期比 `huanxiduo.cn` 更省事 |
| `care.icloush.com` | Care 品牌站 | 静态/SSR 页面 | 先承接服务介绍与线索收集 |

如果后续您坚持让 `huanxiduo.cn` 或 `icloushlab.com` 直接对外承载正式站点，也不是不能做，但那会把备案工作从“围绕一个主域名组织”升级成“围绕多个顶级域名分别核验与准备”，复杂度会明显上升。因此从当前时点看，**子域名统一承载是最适合的生产起步方案**。

## 五、从服务器下单后到可提交备案前的基础初始化顺序

既然服务器方案已经确认，下一个关键问题就是：**哪些事应该先做，哪些事必须等备案通过后再做。** 这里最容易踩坑的是，一买完服务器就急着把正式主域名解析到大陆实例，并直接对外开放。正确顺序应当是“先准备、后提审、再切流”。

| 顺序 | 动作 | 目标 | 是否应立即执行 |
| --- | --- | --- | --- |
| 1 | 完成轻量服务器下单并确认购买时长 | 满足备案资源门槛 | 是 |
| 2 | 检查实例公网 IP、地域、计费方式 | 确认与备案接入条件一致 | 是 |
| 3 | 初始化系统：创建运维用户、更新补丁、关闭密码登录、启用 SSH Key | 提升基础安全性 | 是 |
| 4 | 预配置 Nginx、Node、PM2、日志目录 | 形成可部署环境 | 是 |
| 5 | 配置安全组，仅放行业务需要端口 | 收紧暴露面 | 是 |
| 6 | 整理备案材料与主体信息 | 为提审做准备 | 是 |
| 7 | 判断备案类型：首次 / 接入 / 新增服务 | 决定走哪条备案流程 | 是 |
| 8 | 在腾讯云备案系统提交申请 | 进入审核链路 | 是 |
| 9 | 备案审核期间，先用测试域名或临时地址验证服务 | 不影响备案合规 | 是 |
| 10 | 备案通过后再正式解析生产域名并签发正式 HTTPS | 完成生产切流 | 否，需待备案通过 |

这里需要特别强调一点：**备案提交前可以做环境初始化和服务部署演练，但不建议将正式生产域名全面指向大陆生产实例并作为正式对外网站开放。** 腾讯云官方也明确指出，若未在对应接入商完成备案，会被识别为未备案域名，不可在该平台上开展业务。[1]

## 六、备案号使用路径的判断方法

您之前提到已经存在一个与“小程序/智慧工厂 ERP 测试”相关的备案号。此时不应简单理解成“官网是不是还要再注册一个新的备案号”，而应先判断这个备案号到底属于什么场景。腾讯云备案文档区分得很清楚：如果主办者与域名此前都从未备案，走**首次备案**；如果主体已在其他接入商完成备案并要迁到腾讯云，走**接入备案**；如果主体已经在腾讯云有备案，再新增网站/域名服务，则通常走**新增服务**。[1]

| 当前实际情况 | 对应备案动作 | 是否通常需要新主体备案号 |
| --- | --- | --- |
| 主体从未备案 | 首次备案 | 是 |
| 主体已备案，但备案在其他接入商 | 接入备案 / 新增服务（原备案不在腾讯云） | 否 |
| 主体已在腾讯云有备案 | 新增服务（原备案在腾讯云） | 否 |
| 主体信息发生变化 | 变更备案 | 否 |

所以，下一步最务实的动作不是问“还要不要新的备案号”，而是做一张**备案资产核对表**：

| 需要核对的字段 | 说明 |
| --- | --- |
| 备案主体全称 | 是否为“上海环洗朵科技有限公司” |
| 已有备案号 | 是否已存在主体级备案记录 |
| 当前备案接入商 | 是否在腾讯云、阿里云或其他服务商 |
| 备案下网站服务名称 | 是否已有官网类网站项 |
| 计划上线域名 | 是否为 `icloush.com` 或其子域 |
| 域名实名持有人 | 是否与备案主体完全一致 |
| 当前解析目标 | 是否已经解析到大陆生产服务器 |

只要这张表核出来，备案路径基本就能一锤定音。

## 七、上线切换与解析顺序建议

从执行层面看，正式上线最怕两类问题：一类是备案没走通就把域名切到大陆生产站；另一类是备案虽然通过了，但证书、Nginx、回源与后台安全策略没准备好，导致一上线就暴露风险。因此，解析与切换最好按固定顺序推进。

| 阶段 | 推荐动作 | 目的 |
| --- | --- | --- |
| 环境准备阶段 | 先不动正式主域解析，使用服务器 IP、临时域名或测试子域验证应用 | 避免备案前误暴露正式站 |
| 备案提审阶段 | 确认 `icloush.com` 作为主备案域，提交主体、域名、云资源信息 | 锁定备案主线 |
| 审核等待阶段 | 完成 Nginx、HTTPS、日志、监控、备份配置 | 为正式切流做准备 |
| 备案通过后 | 先解析 `icloush.com` 与 `www.icloush.com`，再逐步放开 `shop`、`admin`、`api` 等子域 | 先主入口，后业务域 |
| 切流后一周 | 观察访问日志、错误率、认证链路和后台登录情况 | 确认稳定性 |

在这个阶段，**后台域名与 API 域名不应和公开品牌域名使用同样的暴露策略**。像 `admin.icloush.com` 这类管理入口，建议至少加上来源限制、WAF 规则或二次认证措施；否则就算备案与证书都正确，生产运维风险依然偏高。

## 八、结合当前项目的明确建议

综合您已经完成的动作和当前项目形态，我的建议可以收束为四句话。

第一，**就按现在的腾讯云轻量服务器方案推进，不需要因为首期上线而再反复犹豫规格选型。** 2 核 4G 在第一阶段足够承担官网、后台与基础接口，但数据库和对象存储应尽量独立。

第二，**把 `icloush.com` 固定为第一主备案域名**，其余品牌优先收敛到 `lab.icloush.com`、`tech.icloush.com`、`care.icloush.com` 等子域；已注册的独立域名先做品牌保护和跳转储备。

第三，**马上核验现有备案号归属**，确认它到底是主体首次备案、其他接入商既有备案，还是腾讯云名下已有备案。这个动作将直接决定您是走首次备案、接入备案，还是新增服务。

第四，**在备案通过前不要贸然把正式域名全面切到大陆生产站**。环境可以先搭，服务可以先测，但正式对外流量切换要等备案状态明朗以后再做。

## 二、腾讯云购买配置单

如果您的目标是尽快搭起第一阶段官网、后台和 API 入口，而不是一次性把所有生产设施做重，那么当前购买策略应偏向“**轻量可上线，但保留升级空间**”。对 iCloush 这种站群 + 后台 + 接口的一体化项目来说，**2 核 4G可以作为首发配置**，前提是把数据库、对象存储、CDN 等外围能力尽量拆出去，不要把一切都堆在这一台机器上。

| 配置项 | 建议值 | 原因与说明 |
| --- | --- | --- |
| 产品类型 | 腾讯云轻量应用服务器 | 适合第一阶段快速上线、部署简单、成本可控 |
| 地域 | **上海** 优先 | 与您当前主体和既有业务场景更一致，沟通与资料准备也更顺 |
| 操作系统 | **Ubuntu 22.04 LTS** 或 **Ubuntu 24.04 LTS** | 建议使用长期支持版，便于后续 Node、Nginx、Docker、PM2 运维 |
| 镜像类型 | **纯净系统镜像** | 不建议选择预装复杂业务环境的镜像，便于后续标准化部署 |
| CPU / 内存 | **2 核 4G 起步** | 适合第一阶段主站、后台、API 轻量并发；后续可升级 |
| 系统盘 | **60GB SSD 或以上** | 便于存放代码、日志、构建产物与临时缓存 |
| 带宽 | **5Mbps 起步** | 官网+后台首发可用；如果后续有较多图片直出或接口并发，可上调 |
| 公网流量 | 选择较宽松套餐 | 避免首发期间因图片、静态资源或调试流量过快耗尽 |
| 登录方式 | **SSH 密钥优先**，密码为备选 | 降低暴力破解风险，便于后续多人协作 |
| 自动续费 | **开启** | 防止服务器因忘记续费导致业务中断 |
| 快照 / 备份 | **开启周期快照** | 用于配置回滚和应急恢复 |
| 安全组 | 仅开放 22、80、443；数据库端口不对公网开放 | 避免把管理端口和数据库直接暴露在公网 |
| 站点入口 | Nginx 统一反向代理 | 统一处理 HTTPS、子域名分发和证书 |
| 应用进程 | PM2 或 systemd 守护 Node 进程 | 保障异常重启和开机自启 |
| 数据库 | **独立云数据库 MySQL** | 不建议把数据库长期放在同一台轻量服务器上 |
| 静态资源 | **COS + CDN** | 降低源站带宽压力，提高页面打开速度 |

从您的截图看，您当前正在看的 **Ubuntu + 上海 + 2 核 4G** 这条路线是正确的。如果只问“能不能买”，答案是**可以直接买**。但如果想把这个配置变成一个更稳的生产起点，我建议您下单时同时坚持以下原则：不要把数据库部署在同一台机器上对外长期服务，不要开放高危端口到公网，也不要直接把测试端口作为正式入口暴露出去。

## 四、域名过户 / 实名一致化清单

您现在最需要处理的，不是“域名能不能先运营”，而是“**备案时域名实名是否能顺利通过核验**”。按照当前已核对的规则，企业备案场景下，域名实名持有人应与备案主体或主体负责人一致；而域名实名信息变更后，通常还需要预留同步时间，再提交备案会更稳妥。[2]

| 核验项 | 应核对内容 | 通过标准 | 当前建议动作 |
| --- | --- | --- | --- |
| 域名持有人 | `icloush.com` 当前实名主体是谁 | 与“上海环洗朵科技有限公司”一致，或至少能证明与主体负责人一致 | **立即核实** |
| 注册商 | 域名所在注册商是否合规 | 注册商需为获批机构 | 保留截图或后台记录 |
| 域名状态 | 是否为正常持有、未锁定异常、未争议 | 状态正常，可操作实名或过户 | 在注册商后台核验 |
| 到期时间 | 距到期是否过近 | 建议保留充足有效期 | 若接近到期，先续费 |
| 历史备案 | 是否存在他人历史备案残留 | 无异常旧备案占用 | 如有异常，先处理旧备案 |
| 过户能力 | 是否能从原公司过户至环洗朵科技 | 能完成企业间过户并更新实名 | **备案前优先完成** |
| 实名同步 | 过户/实名变更后是否已同步到备案系统 | 留出 2–3 天同步窗口更稳妥 | **不要当天变更当天备案** |

如果 `icloush.com` 当前确实在另一家公司名下，但您又希望由“上海环洗朵科技有限公司”作为正式官网和大陆云服务器的备案主体，那么最稳的做法不是“先试试看能不能提交”，而是**先把域名持有人统一过去**。这样做的直接好处是，后续无论是网站备案、接入备案、增补材料、证书申请还是品牌资产归档，都更顺。

## 六、ICP 备案办理顺序表

对您当前这个项目，真正高效的推进方式不是“先把所有域名都备案完”，而是**先用 `icloush.com` 作为第一阶段统一主域，把官网、后台、客户中心、API 收拢到同一主域及子域体系下**。因为主域备案完成后，其下子域一般无需再次单独备案，这会显著降低前期备案量。[2]

| 阶段 | 动作 | 执行目标 | 负责人建议 |
| --- | --- | --- | --- |
| 第 1 步 | 确认备案主体为“上海环洗朵科技有限公司” | 统一全部后续官网与系统备案归口 | 业务负责人 |
| 第 2 步 | 确认腾讯云为中国大陆接入商 | 固定服务器接入商与备案提交入口 | 业务负责人 / IT |
| 第 3 步 | 核实 `icloush.com` 当前实名主体 | 判断是否需要过户或实名修正 | 域名管理员 |
| 第 4 步 | 如不一致，则完成域名过户或实名一致化 | 通过域名核验前置条件 | 域名管理员 |
| 第 5 步 | 等待实名信息同步 | 降低备案提交被退回概率 | 域名管理员 |
| 第 6 步 | 采购腾讯云轻量服务器并保留实例信息 | 为备案提供接入资源 | IT |
| 第 7 步 | 判断备案类型：新增网站备案或接入备案 | 避免走错流程 | IT / 备案负责人 |
| 第 8 步 | 提交备案资料 | 进入正式审核流程 | 备案负责人 |
| 第 9 步 | 备案通过后再做正式 DNS 切换 | 避免未备案先正式上线的风险 | IT |
| 第 10 步 | 配置 SSL、Nginx、WAF、正式发布 | 完成生产上线 | IT / 开发 |

## 七、您的场景下如何判断“新增网站备案”还是“接入备案”

这一步最容易混淆，我把您的情况直接写成判断表。由于您已经有一个主体备案基础，后续大多数情况并不是重新开全新主体备案，而是围绕该主体继续新增或接入。[1] [3]

| 当前情况 | 应优先判断的动作 | 说明 |
| --- | --- | --- |
| “上海环洗朵科技有限公司”已有备案主体，`icloush.com` 准备作为新官网主域，且该主域尚未在该主体名下办理网站项 | **新增网站备案 / 新增服务项** | 最常见情形 |
| 现有已备案业务原来接在别的大陆云厂商，现在准备迁到腾讯云 | **接入备案** | 不属于重新首次备案[3] |
| 同一主域备案通过后，新增 `admin.icloush.com`、`api.icloush.com`、`care.icloush.com` 等子域 | **通常不需单独再备案** | 依附主域备案使用[2] |
| 未来启用 `icloushcare.com` 这类新的独立顶级域名 | **需重新判断是否新增网站项** | 因为是新的主域名，不等同于子域 |

## 十、我给您的直接执行建议

如果您希望把事情推进得又快又稳，我建议就按下面这条顺序往前走。**今天可以先完成腾讯云服务器下单；明天优先核实 `icloush.com` 当前实名主体；如果不一致，就立刻发起过户或实名修正；等同步完成后，再以“上海环洗朵科技有限公司”为主体在腾讯云提交对应网站备案或接入备案。**
